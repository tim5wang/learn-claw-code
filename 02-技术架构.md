# 02 - 技术架构

## 🔄 技术演进

### 第一阶段：Python 原型

```
src/
├── __init__.py
├── commands.py          # 命令端口元数据
├── main.py              # CLI 入口点
├── models.py            # 子系统/模块/待办事项数据类
├── port_manifest.py     # Python 工作区结构摘要
├── query_engine.py      # 渲染 Python 端口摘要
├── task.py
└── tools.py             # 工具端口元数据
```

**CLI 命令：**
```bash
python3 -m src.main summary      # 渲染 Python 端口摘要
python3 -m src.main manifest     # 打印 Python 工作区清单
python3 -m src.main subsystems --limit 16  # 列出 Python 模块
python3 -m src.main commands --limit 10    # 检查命令清单
python3 -m src.main tools --limit 10       # 检查工具清单
python3 -m src.main parity-audit           # 与本地存档进行奇偶校验
```

## 🏗️ 核心架构

### Harness 工程理念

claw-code 的核心是研究 **Agent 系统如何：**
1. 连接工具（Tool Wiring）
2. 编排任务（Task Orchestration）
3. 管理运行时上下文（Runtime Context Management）

### 与原始 Claude Code 的关系

| 方面 | 原始 Claude Code | claw-code |
|------|------------------|-----------|
| **源码** | 泄露的 TypeScript | 从零重写的 Rust/Python |

## 🔧 功能模块

### 命令系统（Commands）

- 端口元数据管理
- 命令清单镜像
- 与原始系统保持奇偶校验

### 工具系统（Tools）

- 工具端口元数据
- 工具清单镜像
- 可扩展工具链

### 任务系统（Task）

- 任务编排
- 持久执行循环
- 验证和完成纪律

### 查询引擎（Query Engine）

- 工作区摘要渲染
- 端口状态追踪
- 奇偶校验审计
